# 影子验证 vs CI 双轨验证 — 对比分析

## 核心区别

| 维度 | CI 双轨验证 | 后台影子验证 |
|------|------------|-------------|
| **目标** | 发现 VM 已支持特性的**语义偏差**（bug） | 发现 VM **尚未支持的特性**（gap） |
| **对比对象** | VM 输出 vs Clang 输出 | Clang 编译成功但 VM 失败的用例 |
| **数据用途** | 修复 bug，建立"标准兼容性"信任 | 指导编译器扩展优先级，缩小子集差距 |
| **频率** | 每次 commit/PR | 每次用例收集迭代（如每周/每月） |
| **输出** | pass/fail + 差异报告 | 缺失特性频率排序 + 优先级建议 |

## 简单比喻

- **CI 双轨验证** = 考试后检查"已学过的知识点有没有答错"
- **影子验证** = 分析考卷发现"哪些知识点还没学过，哪些最常考"

## 互补性

两者不是二选一，而是**分层协作**：

```
CI 流水线
    │
    ├─ 第一层：影子验证（收集缺失特性数据）
    │      └─ 输出：《缺失特性频率报告》
    │      └─ 驱动：ROADMAP 优先级调整
    │
    └─ 第二层：双轨对比（验证已支持特性正确性）
           └─ 输出：《语义偏差报告》
           └─ 驱动：修复 VM bug
```

## 为什么影子验证更适合当前阶段

当前 Cide 的状态：
- 已支持特性（float/指针/struct/数组等）**基本稳定**
- 缺失特性（double/函数指针/文件IO等）**是主要矛盾**

此时做纯双轨对比，大部分用例会卡在"编译失败"（因为特性未实现），无法进入"输出对比"阶段。

影子验证先把这些"编译失败"的用例**分类统计**，告诉你：
- "double 类型导致 28% 的用例失败"
- "函数指针语法导致 19% 的用例失败"
- "文件 I/O 导致 15% 的用例失败"

这比单纯说"测试失败了"更有行动力。

## 融合方案推荐

```
阶段 1：影子验证（立即做，1 周）
  └─> 收集 200+ 用例的 Clang vs Cide 结果
  └─> 生成分类频率报告
  └─> 确定 Top 3 缺失特性 → 进入开发

阶段 2：双轨对比（持续做，每周）
  └─> 对已支持的特性，持续对比 VM vs Clang 输出
  └─> 确保扩展新特性时不引入回归
```
