# Cide 移动端教学可视化受限编辑器：三语言服务拓展计划

> 记录日期：2026-06-28
> 核心定位：移动端教学可视化受限编辑器
> 服务对象：C 教学子集、C++ 教学子集、信语言
> 核心理念：一切白箱操作

---

## 1. 定位与目标

### 1.1 Cide 是什么

Cide 是**面向教学场景的移动端可视化受限编辑器**，不是通用 IDE、不是生产编译器、不是 Clang 替代品。

三个关键词：

- **移动端**：主要运行环境是 Android 平板/手机，次要 Desktop Windows。学生可以随时随地写代码、看执行、学算法。
- **教学可视化**：程序执行过程必须可见、可解释、可单步。内存、指针、栈、堆、算法步骤、字节码都是教学素材。
- **受限编辑器**：只支持明确划定的语言子集。不支持完整 C17/C++20/信全特性。超出子集的代码给出清晰诊断，引导学生写出"教学上正确"的代码。

### 1.2 三语言服务目标

Cide 同时为三种语言提供**一致的教学体验**：

| 语言 | 输入形式 | 执行路径 | 目标用户 |
|------|---------|----------|----------|
| C | `.c` 源码 | Cide C 子集 → 字节码 → CideVM | C 语言初学者 |
| C++ | `.cpp` 源码 | Cide C++ 子集 → 字节码 → CideVM | C++ 教学（class/模板/RAII） |
| 信 | `.信` 源码 | 信编译器 → C17 代码 → Cide C 子集 → 字节码 → CideVM | 中文编程教学用户 |

三语言在 Cide 中共享：

- 同一套编辑器（CideEditor）
- 同一套编译管线（Rust native）
- 同一套虚拟机（CideVM）
- 同一套调试/可视化面板（内存、输出、变量、算法步骤）
- 同一套诊断风格（错误码 + 修复建议 + 知识卡片）

### 1.3 白箱操作承诺

"白箱"意味着学生看到的不是黑盒结果，而是程序运行的内部机制：

1. **源代码可见**：编辑器、语法高亮、错误定位。
2. **编译过程可见**：词法 → 语法 → 类型检查 → 字节码，每一步都可查看。
3. **执行过程可见**：单步执行、断点、调用栈、变量值。
4. **内存可见**：栈帧、堆分配、指针指向、 freed 状态、碎片。
5. **算法可见**：预定义步骤语义、教学描述、指针追踪动画。
6. **运行时可见**：Bytecode Libc 实现的函数可被 step into，不是黑盒。

为实现白箱，**优先使用 Bytecode Libc 而非 Host Func**。Host Func 只用于无法或不宜用 Cide 子集实现的能力（如浮点数学库 `sin`/`cos`，或性能敏感操作）。

## 2. C/C++ 子集拓展计划

### 2.1 已支持能力（基线）

Cide 当前已支持的教学子集详见 `docs/current/C_SUBSET_SPEC.md` 与 `AGENTS.md`，包括：

- 标量类型、数组、VLA、指针、结构体、联合体、枚举、typedef
- 控制流、函数、递归、函数指针
- malloc/free/realloc、VFS 文件 I/O、常用 libc 子集
- `#define` 宏、标准库存根
- C++ class/继承/模板/RAII/引用/unique_ptr（Stage 0~6）

### 2.2 本次必须补齐的 C/C++ 特性

为支持信语言生成的 C17 代码，同时提升 C/C++ 教学完整性，需要补齐以下特性：

#### P0：指针复合赋值（高优先级，低工作量）

| 特性 | 语义 | 改造模块 |
|------|------|----------|
| `p += n` | 指针按 `sizeof(*p)` 前进 n 个元素 | TypeChecker、BytecodeGen |
| `p -= n` | 指针按 `sizeof(*p)` 后退 n 个元素 | TypeChecker、BytecodeGen |
| `void* p += n` | 按 1 字节处理（GCC/Clang 扩展） | TypeChecker、BytecodeGen（void* 步长=1） |
| 多级指针 `pp += n` | 按 `sizeof(T*)` 缩放 | TypeChecker、BytecodeGen |
| 右侧副作用 `p += i++` | 与整数复合赋值一致 | 无需额外改动 |

**验收标准**：新增 baseline 用例通过 Shadow Verification，错误场景诊断准确。

**白箱影响**：指针移动可视化已在 `PointerSnapshot` 中支持，无需新增。

#### P1：`_Generic` 泛型选择（中优先级，中等工作量）

C11 特性，信用它做输出类型分发：

```c
#define _信书元(x) _Generic((x), \
    int64_t: _信书整, \
    信文:    _信书文, \
    浮:      _信书浮  \
)(x)
```

| 改造点 | 说明 |
|--------|------|
| Lexer | 新增 `_Generic` 关键字 token |
| Parser | 新增 `GenericSelection` 表达式节点 |
| TypeChecker | 按控制表达式类型匹配关联表达式，支持 `default` |
| BytecodeGen | 只生成匹配分支的字节码 |
| 测试 | `baseline/c11_generic.c` |

**验收标准**：`_Generic(1, int: 10, default: 20)` 返回 10；`_Generic("hi", char*: 1, default: 0)` 返回 1。

#### P2：复合字面量（中优先级，中高工作量）

C99/C11 特性，信用它构造结构体：

```c
信文 s = (信文){ .字 = "hello", .长 = 5, .容 = 5 };
```

| 改造点 | 说明 |
|--------|------|
| Parser | 在 primary expression 中识别 `( type-name ) { initializer-list }` |
| AST | 新增 `CompoundLiteral` 节点，语义为 lvalue |
| TypeChecker | 按目标类型检查初始化列表 |
| BytecodeGen | 在栈/临时区分配存储，初始化，留下地址 |
| 测试 | `baseline/compound_literal.c` |

**验收标准**：`struct S s = (struct S){1, 2}; printf("%d", s.x);` 输出 1。

#### P3：`__attribute__((cleanup(...)))` 的替代方案（低优先级）

信用 GCC 扩展 `cleanup` 实现 `持…后…` 资源管理。Cide 教学子集**不建议直接支持 GCC attribute**，因为：

- 不是 C 标准。
- 与 Cide 现有 C++ RAII/scope exit 机制重复。
- 增加 Parser 复杂度。

**推荐方案**：

1. 信移动端后端关闭 `cleanup` 生成（改为显式释放或信自己的 RAII lowering）。
2. 或者信的 C17 输出中用显式的 `try/finally` 风格（如果信允许）。
3. Cide 不新增 attribute 支持，但记录此限制。

### 2.3 可选增强（后续按需）

| 特性 | 教学价值 | 工作量 | 建议 |
|------|---------|--------|------|
| 柔性数组 `struct { int n; int a[]; }` | 中 | 中 | 后续按需 |
| `_Static_assert` | 低 | 低 | 可跳过，教学中少用 |
| `_Alignas` / `_Alignof` | 低 | 中 | 可跳过 |
| 匿名结构体/联合体成员（C11） | 中 | 中 | 后续按需 |
| 复数类型 `_Complex` | 低 | 高 | 不支持 |
| 原子类型 `_Atomic` | 低 | 高 | 不支持 |

### 2.4 诚实记录边界

所有不支持或行为有差异的特性必须记录在案：

- `void*` 指针算术按 1 字节处理（GCC/Clang 扩展，非标准 C）。
- 函数指针算术不支持。
- `__attribute__` 不支持（信需关闭 cleanup 生成）。
- `_Generic` 默认分支匹配规则与 C11 一致。
- 复合字面量生命周期到块结束（与 C 标准一致）。

## 3. 信语言接入计划

### 3.1 接入原则

信语言**不直接编译到 Cide 字节码**，而是：

```
.信 源码 → 信编译器 → C17 代码 → Cide C 子集 → Cide 字节码 → CideVM
```

这样信后端保持现有 C17 输出不变，Cide 负责把 C 子集补到能执行这些代码。

### 3.2 信运行时库移植

信运行时包括：字符串（`信文`）、列（`信列`）、结果类型（`信应`）、输入输出辅助函数等。

**首选方案：Bytecode Libc（白箱）**

用 Cide 支持的 C 子集实现信的运行时：

```c
// runtime/xin/信文.c
#include <stdlib.h>
#include <string.h>

typedef struct {
    char* 字;
    long long 长;
    long long 容;
} 信文;

信文* 信文从字(const char* s) {
    信文* p = (信文*)malloc(sizeof(信文));
    long long len = 0;
    while (s[len]) len++;
    p->字 = (char*)malloc(len + 1);
    // strcpy...
    p->长 = len;
    p->容 = len + 1;
    return p;
}
```

优点：
- 白箱：学生可以 step into 字符串/列的实现。
- 教学价值：展示中文编程语言底层如何用 C 实现。
- 不增加 Rust native 复杂度。

缺点：
- 实现工作量较大。
- 性能比 Host Func 差。

**次选方案：Host Func（性能）**

把信运行时函数用 Rust 实现，注册为 CideVM host func。适用于：
- 性能敏感的核心操作。
- 需要调用操作系统能力（如文件、网络，虽然教学场景少）。

**推荐混合策略**：

| 运行时组件 | 实现方式 | 理由 |
|-----------|----------|------|
| 字符串创建/释放/连接 | Bytecode Libc | 教学可视化价值高 |
| 列的增删查 | Bytecode Libc | 数据结构教学核心 |
| 输入输出格式化 | Host Func 或 Bytecode Libc | 输出性能敏感，可先用 Host Func 闭环 |
| 数学函数 | Host Func | 直接复用 libm |

### 3.3 前端支持

| 功能 | 实现方式 |
|------|----------|
| 文件类型 `.信` | IDE 文件浏览器识别 |
| 语法高亮 | 简单关键字高亮（取、乃、也、毕、述、入、得等） |
| 运行按钮 | 调用 `rust.compileXin(source)` 或先由信编译器生成 C 再调用 `rust.compileC` |
| 错误定位 | 信编译器错误直接定位到 `.信` 行号；Cide 报错定位到生成的 `.c` 行号，再反向映射 |
| 可视化 | 复用 C/C++ 的内存、算法、指针面板 |

### 3.4 诊断映射

信 → C17 的编译错误由信编译器处理，Cide 只需透传。

Cide 对生成的 C17 代码报错时，需要把错误位置映射回信源码。有两种策略：

1. **策略 A：行号注释映射**
   信编译器在生成的 C 代码中插入 `#line` 或注释，Cide 报错时提取原始行号。

2. **策略 B：源码映射表**
   信编译器输出 `.信` 到 `.c` 的 SourceMap 文件，Cide 根据映射表跳转。

**推荐策略 A**，因为 Cide 当前诊断系统基于 `SourceLoc`，`#line` 预处理指令如果支持的话可以直接利用。

### 3.5 最小闭环验证

第一个跑通的信程序：

```信
取「甲」乃十也。
以「书」入「甲」也。
```

预期生成的 C17 代码（简化）：

```c
#include <stdio.h>
#include "信运行时.h"

int main() {
    long long 甲 = 10;
    _信书元(甲);
    printf("\n");
    return 0;
}
```

Cide 执行后输出：`10`。

## 4. 三语言统一体验设计

### 4.1 统一编辑器

CideEditor 需要支持三种语言的差异化但一致的体验：

| 能力 | C | C++ | 信 |
|------|---|---|----|
| 语法高亮 | ✅ 完整 | ✅ 完整 | ⚠️ 基础关键字 |
| 自动补全 | ✅ 上下文感知 | ✅ 上下文感知 | ⚠️ 基础补全 |
| 错误波浪线 | ✅ | ✅ | ✅（透传信编译器错误） |
| 代码模板 | ✅ 大量 | ✅ C++ 模板 | ⚠️ 信模板 |
| 教程引导 | ✅ | ✅ | ⚠️ |

### 4.2 统一编译管线

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  C 源码      │     │  C++ 源码    │     │  信源码      │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                   │                   │
       ▼                   ▼                   ▼
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ Cide C 子集  │     │ Cide C++ 子集│     │ 信编译器      │
│ 编译器       │     │ 编译器       │     │（生成 C17）   │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │                   │                   │
       └─────────┬─────────┴─────────┬─────────┘
                 ▼                   ▼
          ┌─────────────┐     ┌─────────────┐
          │ Cide 字节码  │     │ 诊断/错误    │
          └──────┬──────┘     └─────────────┘
                 ▼
          ┌─────────────┐
          │   CideVM    │
          └──────┬──────┘
                 ▼
          ┌─────────────┐
          │ 可视化/输出  │
          └─────────────┘
```

### 4.3 统一调试与可视化

三语言共享同一套面板：

| 面板 | 内容 |
|------|------|
| 内存映射 Canvas | 栈、堆、全局区、指针箭头 |
| 变量面板 | 局部变量、全局变量、结构体展开 |
| 调用栈 | 函数调用链 |
| 输出面板 | stdout/stderr |
| 算法步骤 | 预定义步骤语义、教学描述 |
| 字节码视图 | 当前执行的 opcode（可选，面向进阶学生） |

### 4.4 统一诊断风格

无论哪种语言，错误信息都遵循：

- 错误码（E3xxx / C++ / XIN）
- 位置（行号/列号）
- 简短描述
- 修复建议
- 知识卡片（关联概念）

信语言的错误由信编译器提供，Cide 负责展示；C/C++ 错误由 Cide 诊断系统提供。

## 5. 白箱操作保障机制

### 5.1 编译期白箱

| 能力 | 实现 |
|------|------|
| Token 流查看 | 编辑器侧边栏展示 Lexer 输出 |
| AST 树查看 | 折叠/展开语法树节点 |
| 类型检查过程 | 展示每个表达式的推断类型 |
| 字节码查看 | 展示生成的 opcode 序列 |

### 5.2 运行期白箱

| 能力 | 实现 |
|------|------|
| 单步执行 | `rust.stepNext()` 逐条 opcode |
| 内存快照 | 每步记录栈/堆状态 |
| 指针追踪 | `PointerSnapshot` 四状态动画 |
| 算法步骤标注 | 27 种预定义算法模板 + 运行时推断 |
| 运行时函数可进入 | Bytecode Libc 函数可被 step into |

### 5.3 受限子集白箱

- 所有不支持的语法明确报错，并说明"Cide 教学子集暂不支持 XXX"。
- 提供替代写法建议。
- 在设置或帮助面板展示当前支持的子集清单。

## 7. 风险与回退策略

| 风险 | 影响 | 回退策略 |
|------|------|----------|
| `_Generic` 实现复杂度超预期 | Phase 1 延期 | 先让信后端用函数重载/宏绕过 `_Generic`，后续再补 |
| 复合字面量 lvalue 语义难以实现 | Phase 1 延期 | 信后端改用临时变量初始化：`struct S __t = {1,2}; 信文 s = __t;` |
| 信运行时 Bytecode Libc 性能太差 | 用户体验差 | 高频函数改用 Host Func，保留 Bytecode Libc 用于教学演示 |
| 信编译器生成的 C17 代码用到其他未支持特性 | Phase 2 阻塞 | 信后端增加 "Cide 兼容模式" 开关，避开这些特性 |
| Flutter 前端支持三种语言工作量过大 | Phase 3 延期 | 先支持命令行 `cide_cli` 跑信程序，前端后续补 |

## 8. 结论

本计划的核心是：**把 Cide 从"C/C++ 教学 IDE"升级为"移动端教学可视化受限编辑器"，同时服务 C、C++、信三种语言**。

关键决策：

1. **不改造信后端**，让信继续输出 C17 代码。
2. **拓展 Cide C 子集**，补齐 `_Generic`、复合字面量、指针复合赋值等特性。
3. **信运行时优先走 Bytecode Libc**，保证白箱教学价值。
4. **三语言共享同一套移动端教学体验**：编辑、编译、调试、可视化、诊断。

这个方案比"信直接生成 Cide 字节码"更务实，比"只用 NDK 跑信"更有教学价值。
