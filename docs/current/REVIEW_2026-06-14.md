## 三、关键问题清单

### 3.1 前端 P0 Bug

#### Bug 1：`LineNumberColumn` 变量访问后缀永远不显示

**文件**: `CideFlutter/lib/editor/gutter/line_number_column.dart:36`

```dart
final markers = ctx.accessedVars
    .where((a) => false) // 由外层统一计算当前行的变量标记
    .take(2)
    ...
```

`where((a) => false)` 导致结果恒为空，变量访问标记功能实际不可用。

#### Bug 2：`applyFix` 基于中文字符串匹配

**文件**: `CideFlutter/lib/providers/ide_notifier.dart:512-639`

Rust 后端已经返回完整结构化修复字段（`fix_kind`、`replace_start_line`、`replace_start_column`、`replace_end_line`、`replace_end_column`、`replacement_text`），但 Dart 端：

1. 结构化替换**只支持单行**（`if (startLine == endLine)`），多行替换直接跳过
2. fallback 使用中文/符号字符串匹配（`fix.contains('分号')`、`fix.contains("}'")` 等），文案微调即失效

**建议**: 将修复逻辑完全下沉到 Rust，新增 `rust.applyFix(filename, source, diag) -> String?` API。

### 3.2 LeetCode 防线当前状态（重要说明）

**现状**: `native/tests/cases/leetcode/` 目录不存在，`LEETCODE_FAILURES.md` 为空白模板。

**原因**: LeetCode 防线是 Phase 28 计划中的一部分（见 `docs/current/PHASE_KR_LEETCODE_TEST_PLAN.md`），但在实际推进过程中，团队优先级调整为集中攻克 C++ 子集扩展（Phase 31+）。C++ 扩展涉及 Lexer/Parser/AST/TypeChecker/BytecodeGen/VM 全管线的重大改造，工作量和教学价值更高，因此 LeetCode 防线**暂时搁置**。

**计划**: LeetCode 防线仍是项目计划的一部分，将在 C++ 子集达到稳定状态后重新启动。当前 `cide_e2e.rs` 中已保留 `test_cide_e2e_leetcode` 和 `test_cide_e2e_leetcode_known_failures` 两个空壳测试，用例目录和 golden 输出待后续填充。

**诚实记录**: 这是项目自述的"记录问题而非粉饰"原则的体现。AGENTS.md 中"防线 2：K&R + LeetCode 真实程序回归（已有）"的描述当前部分不准确，建议在 AGENTS.md 中明确标注 LeetCode 为"计划中，待 C++ 子集稳定后实施"。

### 3.4 `cargo fmt` 实际不可用

`rustfmt.toml` 配置了 4 个 nightly-only 特性：

```toml
group_imports = "StdExternalCrate"
imports_granularity = "Module"
wrap_comments = true
format_code_in_doc_comments = true
```

CI 和本地都使用 stable toolchain，导致这些特性不生效且每次 `cargo fmt` 打印警告。`cargo fmt --check` 还会检测出 4 处格式差异：

- `native/src/bin/cide_cli.rs:293`
- `native/src/capi/mod.rs:303`
- `native/src/compiler/algorithm_detector.rs`（换行风格）
- `native/src/compiler/ast.rs:578, 871`

**建议**: CI 改用 nightly toolchain 运行 `cargo fmt --check`，或移除 nightly-only 配置。

### 3.5 类型重复定义

| 类型 | 定义位置 1 | 定义位置 2 | 定义位置 3 |
|---|---|---|---|
| `FuncMeta` | `session.rs:41` | `compiler/codegen/mod.rs:25`（多 `return_type`） | `vm/vm/mod.rs:35` |
| `Symbol` / `VMSymbol` | `session.rs:52`（`Symbol`） | - | `vm/vm/mod.rs:68`（`VMSymbol`） |

| `base_kind` | `compiler/codegen/mod.rs:7` | `vm/vm/mod.rs:26` | - |

`compile_pipeline.rs` 需要在三种 `FuncMeta` 之间手动转换，容易遗漏字段。

**建议**: 定义 `shared::FuncMeta` 和 `shared::Symbol`；将 `base_kind` 提升为 `Type` 的方法；统一 `SourceLoc`。
